Modular CSS

The purpose of modularizing our CSS is to help it become much more flexible and scalable, but also to help make it easier to read and easier to maintain.
Focusing on creating healthy front-end modules instead of complete pages can help break complex page layouts into reusable solutions.
— Dave Rupert
Dave Rupert says it best when he says that we should focus on creating front-end modules as reusable solutions for complex pages.
This isn’t the “homepage” of Apple.
It's a complex page fit together with front-end modules. It's a collection of reusable style patterns.

small pieces
independent
portable

Modular CSS is all about learning to think about your CSS in terms of individual pieces. Each Module should be designed to exist as a standalone component. In doing so, the page will be more flexible. If done right, Modules can easily be moved to different parts of the layout without breaking. Breaking your CSS down into smaller chunks will ultimately result in more maintainable code.

SMACSS

Scalable & Modular Architecture for CSS

SMACSS stands for Scalable and Modular Architecture for CSS.
It was created by Jonathan Snook of Shopify. The SMACSS core goals are to increase the semantic value of a section of HTML and content while decreasing the expectation of a specific HTML structure [next]

Framework

It is not a framework, but an approach to authoring stylesheets. It doesn’t force you to do thins certain way, and it doesn’t have any dependencies. If you don’t like the way something is done, change it or leave it out. [next]

categorization
naming conventions
depth of applicability

SMACSS has three basic principles: categorization, naming conventions, and depth of applicability (which is really hard to say).

Categorization

The basic concept of SMACSS is to categorize styles into 5* categories:
  • base
  • layout
  • modules
  • states
  • themes
The basic concept of SMACSS is to categorize styles into 5* categories: Base, layout, modules, states & themes.  Each category comes with a set of more or less loosely defined naming conventions and usage rules. [next]

Base
CSS resets, default styles
(ex. html, body, h1, ul, etc)

Base rules are the defaults. They are almost exclusively single ele- ment selectors but it could include attribute selectors, pseudo-class selectors, child selectors or sibling selectors. Essentially, a base style says that wherever this element is on the page, it should look like this.

Layout
grid, major components
ex. header, sidebar, nav

Layout rules divide the page into sections. Layouts hold one or more modules together.Wrappers, rows, columns and any grid-based styles can go here. Also major semantic areas as well, such as header, content and footer. - When defining a layout rule make sure you don’t mix presentation rules such as fonts, colors, backgrounds or borders. - Layout rules should only contain box-model properties like margins, padding, positioning, width, height, etc.,
Let’s look go back to Apple. A quick analysis of the home page, you can see some defined areas.

Modules
content patterns
(ex. search-box, navigation, content-box)

Modules are the reusable, modular parts of our design. They are the callouts, the sidebar sections, the product lists and so on.
The amount of abstraction you use is up to you, but the point is to break you layout down in to as many reusable components as possible.

States
module in various states

State rules are ways to describe how our modules or layouts will look when in a particular state. Is it hidden or expanded? Is it active or inactive? How does it look on different devices? I like to add print styles into this category as well.

Themes
module in various contexts

Theme Rules describe the appearance of modules, i.e., colors, images, etc, by way of visual themes. I’ve never needed to use them, but some context examples might be: user-selected skin for an application, client branding for a hosted site, holiday or seasonal themes.

Naming Conventions

The basic concept of SMACSS is to categorize styles into 5* categories:

Base
h1, h2, p, a, etc.

HTML element selectors

Layout
.layout-*, .l-*

All selectors in this category should be prefixed with .layout- (e.g. .layout-header, .layout-sidebar).

Module
.<name>

These will vary based on what the project needs, so just try to name them something that’s abstract enough to use elsewhere, but still descriptive of the object.

Module
.button

Sub-module
.<name>-<state>

Sub-module
.button-secondary

State
.is-*

State rules are prefixed with “is-”, followed by a name to describe its state of being. Some example names would be “active”, “collapsed”, or “hidden”. So, the namespace makes a statement as if to say, “This ‘is-active’.”

Theme
.theme-*

Jonathan suggests using .theme- for namespacing, but I’m going to disagree with him.

Theme

Depth of Applicability

The basic concept of SMACSS is to categorize styles into 5* categories:
header#top-menu > nav > ul > li a + div > ul > li > ul li:hover > a {…}
Looking at this I bet you know exactly what the HTML looks like.
Each time your go deeper into the markup, you add a generation. This selector effects 10 generations.
header#top-menu > nav > ul > li a + div > ul > li > ul li:hover > a {…}
This is not a module. This is a maintenance nightmare. If anything about the HTML changes, you’re going to have problems. Full disclosure: this is from a dropdown menu I made before I knew any better. Please don’t judge me. There’s no need to use such a specific selector… [next]
header#top-menu > nav > ul > li a + div > ul > li > ul li:hover > a {…}
…when this is really what we care about.
.nav-subitem
Instead, use a single class to name the module or submodule. [pause] [next]
.nav-subitem > a
Or even this. Child selectors are great when used sparingly within modules. Sure, this still relies on a certain HTML structure, but it’s very shallow, and will most likely be portable. And it’s a great way to avoid classitis in your markup.

naming conventions
class selectors
child selectors

Use child selectors — to limit the scope of selectors to direct descendants instead of all descendants Use class selectors — to reduce the need for descendant selectors of any kind Use naming conventions — for more for readable code

Namespacing

The first way Sass can optimize SMACSS is through namespacing. [next]
The ampersand operator is essential in streamlining namespacing in Sass. Basically the ampersand is a way to merge selectors. [next]
.btn {
  &:hover {…}
}
The parent element takes the place of the ampersand. The most common use of this was for pseudo classes such as :hover, :before and :after. [next]
.btn:hover {…}
.btn {
    form & {…}
}
form .btn {…}
This is good, but it’s not ideal for modular css and naming conventions. [next]

&- or &_

Before Sass 3.3, you could only use ampersands in these ways. But now you can now write &-suffix (or &_suffix, or even &suffix if you really want) and Sass will make it work. This is perfect for namespacing your selectors. [next]
.btn {
    &-secondary {…}
    &_secondary {…}
}
Now you can nest your submodules in your main module… [next]
.btn-secondary {…}
.btn_secondary {…}
…and it will compile to the naming conventions I went over earlier. [pause] [next]

< 3 levels deep

Speaking of nesting, it’s generally considered bad form to nest too deeply in Sass, since it can result in the nightmare selector I showed you before. [next] The general rule is not to go more than three level deep. [next]

    &-secondary {
        &-icon {…}
    }
}
But which the recent changes in how the ampersand gets parsed, you can nest as much as you please and not have to worry too much about deep selector chains. [next]
.btn-secondary {…}
.btn-secondary-icon {…}

Selector Inheritance

.btn {…}
.btn-large {…}

<div class='btn btn-large'>
With the SMACSS methodology, you should always add styles to selectors, and not undo them. One side-effects of this is the need to use multiple classes in your HTML, which can get out of hand quickly. I like squeaky clean HTML, so I try to use as well classes as possible. One solution is to… [next]

@extend
all the
things!

Fortunately, using Sass @extend can solve this. [next]
.btn {…}
.btn-large {@extend .btn;}

<div class='btn-large'>
Extending primary modules into submodules utilizes Sass’s selector inheritance, allowing submodules to inherit the properties of the parent selector. The result selectors with common styles are grouped together, and unique properties are outputted separately. [next]
%btn {…}
.btn-large {@extend %btn;}

<div class='btn-large'>
Alternatively, you can use placeholder selectors for your primary modules and extend them within the submodules, which would reduce the number of overall selectors in your CSS output. The primary selector isn’t complied, instead its properties are included in all the submodules. This might not work if you need to style primary modules separately from submodules. [next]

don’t @extend between modules

Like many things with Sass, don’t overdo it. Limit your extends to inside individual modules, otherwise they’ll become overly dependent on either, which limits their portability. [pause] [next]

01. Utilities

utilities/_index.scss

@import 'global';
@import 'functions';
@import 'mixins';
@import 'helpers';

Variables, mixins, functions, etc. Basically anything that doesn’t output CSS by itself.

02. Libraries

utilities/_lib.scss

@import "lib/susy";
@import "lib/font-awesome";
@import "lib/pesticide";

Third-party libraries such as Susy, Font Awesome, Pesticide, and other plugins.

For JS plugins that require CSS files, just change the extension to .scss and import it into your stylesheet.

03. Base

base/_index.scss

CSS resets, Normalize, element styles

Theming

when using SASS, theming can easily be handled by defining variables for the style properties that you want to be themeable (e.g. $border-color).
@mixin theme($name) {
  @if $theme == $name {
     @content; } }
First, create use a mixing with nested if statements you can use to define theme styles within your modules.
$theme: rebeccapurple
Then, declare a variable for the theme you want to create.
@include theme($rebeccapurple)
Lastly, include the mixin with the defined variable in your module.
